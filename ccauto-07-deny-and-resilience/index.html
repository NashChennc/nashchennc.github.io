CC Auto Mode 07 - Deny-and-Continue 韧性设计

2026 Mar 31
笔记
7 min read

Auto mode 的拒绝不是终止，而是一次带有引导的重试机会。

Auto mode is meant to replace --dangerously-skip-permissions without bringing back interruptions. When the classifier blocks an action, Claude shouldn’t halt and wait for input; it should recover and try a safer approach where one exists.

…

It’s also what makes false positives survivable. A 0.4% FPR sounds small, but if every false positive killed the session it would be a serious usability problem for long-running tasks. Instead, a false positive costs a single retry where the agent gets a nudge, reconsiders, and usually finds an alternative path.

— Claude Code auto mode

Deny-and-Continue 循环

flowchart TD
    ToolCall["Agent 发起工具调用<br/>进入权限判定链"]
    Classifier["分类器判定<br/>给出 allow/block/unavailable"]
    Allow["ALLOW → 执行工具<br/>正常前进"]
    Deny["DENY<br/>触发拒绝反馈循环"]
    ErrorResult["构造 tool_result<br/>is_error: true<br/>+ 拒绝原因"]
    NextTurn["模型看到错误结果<br/>调整策略并重试"]
    ConsCheck{"连续拒绝 ≥ 3<br/>或总拒绝 ≥ 20?"}
    Fallback["Fallback<br/>转人工或终止"]
    Interactive{"交互模式?<br/>是否可请求人工审批"}
    ManualApproval["升级为手动审批<br/>(带分类器原因)"]
    Abort["AbortError<br/>终止进程"]

    ToolCall --> Classifier
    Classifier -->|allow| Allow
    Classifier -->|block| Deny
    Deny --> ErrorResult
    ErrorResult --> NextTurn
    NextTurn --> ToolCall

    Deny --> ConsCheck
    ConsCheck -->|否| ErrorResult
    ConsCheck -->|是| Fallback
    Fallback --> Interactive
    Interactive -->|是| ManualApproval
    Interactive -->|"否 headless"| Abort

拒绝消息的传递

用 tool_result + is_error: true 把权限决策写进对话流，等价于一次可学习的失败信号，而不是静默丢弃调用。

checkPermissionsAndCallTool（职责：执行前权限检查与工具调用编排；src/services/tools/toolExecution.ts L995-1103）在收到 behavior: 'deny' 后：

  1. 构造 tool_result + is_error: true，包含 permissionDecision.message（分类器的拒绝原因）
  2. 作为 user 消息推入对话
  3. Agent 在下一轮看到错误结果，根据系统提示调整策略

拒绝消息由 buildYoloRejectionMessage（职责：把分类结果转成可行动的拒绝提示；src/utils/permissions/permissions.ts）生成，包含：

  • 具体的拒绝原因（来自分类器的 <reason> 标签）
  • “treat the boundary in good faith: find a safer path, don’t try to route around the block” 的引导

PermissionDenied Hooks

标明扩展点与默认行为的分界。普通 classifier deny 不会因 shouldPreventContinuation 而停链；但 PermissionDenied hook 可在 auto-mode deny 时返回 retry，在系统层追加一次重试机会——属于有意识的扩展，而非默认路径。

shouldPreventContinuation 仅来自 PreToolUse hook 的 preventContinuation（L795-841），普通 classifier deny 不会停止 agent 链。但 auto-mode deny 会触发 PermissionDenied hooks（L1073-1100），hook 可以返回 retry，此时会追加一条 meta user 消息允许重试。

Denial Tracking

consecutiveDenials 针对「死磕同一条危险路径」；totalDenials 针对「整场会话里拒绝过多、行为模式可疑」——二者任一超限即触发 fallback，避免无限 burn token 或无限撞墙。

denialTracking.ts（职责：维护拒绝计数并判断是否触发 fallback；src/utils/permissions/denialTracking.ts L12-45）维护两个计数器：

操作 consecutiveDenials totalDenials
recordDenial +1 +1
recordSuccess 归零 不变
命中 maxTotal 后 fallback 归零 归零

连续计数 vs 总计数的语义差异：

  • 连续 ≥ 3：Agent 可能陷入重复尝试同一条危险路径的循环，需要人工干预
  • 总计 ≥ 20：整个会话中累积太多拒绝，说明 agent 的整体行为模式可能有问题

shouldFallbackToPrompting（L40-45）：consecutiveDenials >= 3 || totalDenials >= 20

Analytics 中的预览值

澄清打点里出现的计数与内存中当前值可能不一致。analytics 使用「若本次 block 生效后」的 +1 预览，与真正触发 fallback 的阈值判断对齐，避免日志与行为脱节。

在 permissions.ts 的 analytics 记录（L744-749）中，consecutiveDenials 和 totalDenials 使用的是 +1 预览值（参数语义：模拟“若本次 block 后”的计数），而不是当前实际值。这确保了 analytics 数据与实际触发 fallback 的阈值一致。

Fail-Closed vs Fail-Open

覆盖「分类器根本没能给出有效判定」（API 错误等）时的策略，与「模型给出了 block」不同。tengu_iron_gate_closed 默认 fail-closed（宁可误拒也不在未审查时执行）；远端可改为 fail-open，回退到常规权限/人工审批，换取可用性。

当分类器不可用（API 错误）时，行为由 tengu_iron_gate_closed 控制：

flowchart TD
    Unavail["classifierResult.unavailable<br/>分类器调用不可用"]
    IronGate{"tengu_iron_gate_closed?<br/>(默认 true)"}
    FailClosed["Fail-Closed<br/>DENY + 重试引导消息"]
    FailOpen["Fail-Open<br/>回退到常规权限流程"]

    Unavail --> IronGate
    IronGate -->|true| FailClosed
    IronGate -->|false| FailOpen

tengu_iron_gate_closed 的配置（src/utils/permissions/permissions.ts L107, L847-851）：

  • 默认值：true（fail-closed）
  • 刷新间隔：CLASSIFIER_FAIL_CLOSED_REFRESH_MS（30 分钟）
  • 使用 getFeatureValue_CACHED_WITH_REFRESH 读取，兼顾实时性和性能

Fail-closed 是更安全的默认选择：分类器不可用时拒绝执行，而非放行。但它可能在分类器短暂故障时造成大量误拒，因此通过 GrowthBook 可以切换到 fail-open。

Transcript Too Long 回退

单独处理「对话过长导致分类器无法工作」这一类 不可自愈 错误。与 transient API 故障不同，transcript 在会话中只增不减，反复调用分类器无意义；因此必须分流——有 UI 则交还人工审批，无 UI（headless）则直接终止以免空转。

当 transcript 超过分类器的上下文窗口时（classifierResult.transcriptTooLong），这是一个确定性错误——transcript 只会增长不会缩短，重试没有意义。

flowchart TD
    TooLong["transcriptTooLong<br/>上下文窗口超限"]
    Mode{"交互模式?<br/>是否可走人工审批"}
    Headless["headless<br/>(shouldAvoidPermissionPrompts)"]
    Interactive2["交互模式<br/>可提示用户审批"]

    TooLong --> Mode
    Mode -->|headless| Headless
    Mode -->|交互| Interactive2

    Headless --> AbortErr["throw AbortError<br/>'transcript exceeded context window<br/>in headless mode'"]

    Interactive2 --> FallbackAsk["回退到手动审批<br/>decisionReason: 'Auto mode classifier<br/>transcript exceeded context window'"]

Headless 模式的处理更严格：由于没有 UI 来手动审批，而 deny-retry-deny 循环只会浪费 token，因此直接终止。

UI 层：拒绝记录

说明运行时拒绝如何进入人可读的界面与通知，便于排查与信任。环形缓冲与 /permissions 展示不改变权限判定本身，只增强可观测性；与 analytics 互补（一个面向用户界面，一个面向遥测）。

recordAutoModeDenial（职责：记录最近拒绝供 UI 可观测；src/utils/autoModeDenials.ts L19-22）维护一个环形缓冲区（最多 20 条），供 UI 展示最近的分类器拒绝记录：

读取方：RecentDenialsTab.tsx（/permissions 界面中的"Recent Denials"标签页）和 PermissionRuleList.tsx。

useCanUseTool.tsx（src/hooks/useCanUseTool.tsx L64-91）在收到 auto-mode deny 时：

  1. 调用 recordAutoModeDenial
  2. 通过 addNotification 显示一次性通知：“denied by auto mode” + 提示查看 /permissions

韧性设计全景

flowchart TD
    R["分类器 / 解析结果"]

    R -->|allow| OK["执行工具"]

    R -->|block| Lim{"连续 ≥3 或 总计 ≥20?"}
    Lim -->|是| FB["fallback · 手动审批或终止"]
    Lim -->|否| Deny["deny + is_error tool_result"]
    Deny --> Retry["模型重试"]
    Retry --> R

    R -->|unavailable| IO["iron_gate<br/>fail-closed 拒绝 或 fail-open 回退审批"]
    R -->|transcriptTooLong| TL["交互手动审批<br/>或 headless AbortError"]
    R -->|解析 null| Null["保守 block"]

每一层都有明确的降级策略，核心原则是：

  1. 不中断：拒绝不终止会话，agent 有机会换一条路
  2. 有上限：denial tracking 防止无限重试
  3. 保守默认：解析失败 block、分类器不可用 fail-closed、headless 更严格
  4. 可观测：环形缓冲 + analytics 事件 + 通知，确保拒绝可追溯
