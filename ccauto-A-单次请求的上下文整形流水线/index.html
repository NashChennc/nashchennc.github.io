CC Auto Mode 附录 A - 单次请求的上下文整形流水线

2026 Mar 31
笔记
15 min read

在长时 Agentic 会话中，随着工具调用（尤其是涉及大量终端输出、文件读取的操作）的不断累积，上下文（Context）不可避免地会迅速膨胀。如果不加干预，这不仅会带来极高的 Token 成本与延迟（效率问题），还会导致模型注意力涣散（质量问题）。

从架构上看，上下文如何被裁剪、汇总与发送还影响第三件事：transcript 过长时分类器等安全组件是否仍可工作（间接安全）。Anthropic 在 Harness design for long-running application development 中讨论了 compaction 与长时 harness；本仓库 并未 实现博文中那种独立的 planner / generator / evaluator 三智能体闭环，但实现了与之相关的 客户端上下文工程（见本篇与 附录 B）。与此同时，Claude Code auto mode 强调 输出层 transcript 分类器 与 PI 探针 的正交互补——权限与分类器细节见 02 — 两阶段分类器管道，多 Agent / AFK / PI 见 04 — 多 Agent 安全交接，Deny 与 transcript 过长降级见 07 — Deny-and-Continue。

为此，Claude Code 在主循环（queryLoop）中设计了一条严格的出站前上下文整形流水线（Context Shaping Pipeline）。这条流水线的核心设计哲学是分层处理与正交组合——将对内容的修改（替换/占位）、对条数的裁剪（Snip），以及对缓存的操作（Microcompact）彻底解耦，按固定顺序依次执行。

在实现上，流水线挂在 queryLoop（src/query.ts 约 L241）的单次迭代内：出站前先用 getMessagesAfterCompactBoundary(messages) 取 compact 边界之后的消息切片（同文件约 L365，函数定义在 src/utils/messages.ts），再依次执行 budget → snip → microcompact →（可选）context collapse → autocompact（同文件约 L369–467，见下文摘录）。

核心术语与坐标系

在深入流水线之前，需要先对齐一个决定上下文如何被处理的核心概念：

  • QuerySource（查询来源）：系统通过该标识判断当前请求的发起者（类型见 src/constants/querySource，以仓库内实际扩展名为准）。
    • 主线程（Main Thread）：例如 repl_main_thread*、agent:*。主线程的请求会影响全局状态，其上下文替换记录会被持久化。
    • 派生分支（Fork / Ephemeral）：例如后台的 session_memory、prompt_suggestion。分支线程的请求是临时性的，严禁写入全局状态或持久化，以免污染主对话上下文。

applyToolResultBudget 前注释（源码摘录）：

Cached Microcompact（缓存编辑）是“内容盲”的，它仅根据 tool_use_id 进行整块摘除；而 Budget 是对 tool_result 正文进行占位符替换。所以 applyToolResultBudget 必须走在 microcompact 之前。如果顺序反转，内容替换就无法与仅按 ID 操作的缓存逻辑干净地组合（Compose cleanly）。

下列摘录与源码注释一致，标出上述顺序在 queryLoop 中的真实落点（行号随版本可能漂移，以符号为准）：

Phase 1: L1 内容层 —— 工具结果预算 (applyToolResultBudget)

这是流水线的第一站。Agent 可能执行了一个返回数万行日志的 Bash 命令，这一层负责将其拦截并“瘦身”。入口函数为 applyToolResultBudget（src/utils/toolResultStorage.ts 约 L924），由 queryLoop 在 microcompact 之前调用（src/query.ts 约 L379）。

如果开启了特性（contentReplacementState 存在），该阶段会执行 enforceToolResultBudget（同文件约 L769），将严重超标的工具输出替换为占位符（Placeholder），告知模型“输出过长已被截断，请使用 grep 等工具精确查找”。

flowchart TD

  noState{state 未定义?}
  retIn["直接返回原 messages（no-op）"]
  enforce["enforceToolResultBudget<br/>对超标 tool_result 写占位内容"]

  skipW[跳过持久化]

  noState -->|是| retIn
  noState -->|否| enforce
  enforce --> replaced

正交设计：“是否发生内容替换”与“是否持久化记录（writeToTranscript）”是完全解耦的。如前所述，只有当 QuerySource 是可恢复的（如主 REPL 或主 Agent）时，才会传入回调将替换动作记录到本地文件中。如果是后台联想（Prompt Suggestion）触发了这套逻辑，即使发生了替换，也绝不会污染磁盘文件。

持久化规则与 applyToolResultBudget 的 JSDoc（toolResultStorage.ts 约 L915–919）一致：writeToTranscript 仅由 query.ts 在 可 resume 的 querySource 上传入；agent_summary、session_memory、/btw、compact 等 临时 fork 路径传 undefined，避免污染侧链或会话记录。

Phase 2: L2 形状层 —— 尾部裁剪 (snipCompactIfNeeded)

在处理完单条超大消息后，系统开始处理消息列表的“形状”。query.ts 在 HISTORY_SNIP 开启时动态加载 services/compact/snipCompact（约 L115–117），入口为 snipCompactIfNeeded（约 L401–404）；图中 L2 标签写作 history snip，与运行时模块名对应。

如果开启了 HISTORY_SNIP，系统会根据策略对对话尾部的消息条数进行轻量级修剪，并计算出因此释放的 Token 数量 (snipTokensFreed)。

这个释放量至关重要：它会作为最后一参传入 deps.autocompact(...)（src/query.ts 约 L454–466）。因为模型返回的 usage.input_tokens 无法反映由于 Snip 刚刚省下的额度，传入 snipTokensFreed 可以修正阈值判断，避免系统产生“上下文依然满载”的幻觉而误触发全量压缩。

Phase 3: L3 压力层 —— 增量与全量收缩

到达这一层，系统开始应对真正的 Context 窗口压力。

  1. Microcompact：一种极其轻量级的“清理”。queryLoop 通过 deps.microcompact 调用，生产绑定为 microcompactMessages（src/services/compact/microCompact.ts 约 L253；deps 装配见 src/query/deps.ts L26–27、34–37）。它尽可能在不破坏整段前缀缓存的前提下，清理掉冗余的 tool_results（此部分涉及极为复杂的全局状态隔离与 Cache Editing 机制，将在《附录 B》中深入解析）。
  2. Context Collapse (可选)：如果开启了 CONTEXT_COLLAPSE，contextCollapse 模块由 src/query.ts 约 L18–19 动态加载，运行时调用 applyCollapsesIfNeeded（同文件约 L440–446）。精妙之处在于：如果这一步折叠后，总 Token 降到了 Autocompact 阈值以下，就会直接跳过后续重量级的重汇总过程，从而为用户保留下更细粒度的近期上下文。
  3. Autocompact：终极防线。deps.autocompact 即 autoCompactIfNeeded（src/services/compact/autoCompact.ts 约 L241）。若估算仍超过 getAutoCompactThreshold(model)（阈值逻辑同文件约 L32–90，含约 13,000 Token 的 AUTOCOMPACT_BUFFER_TOKENS 缓冲），则正式触发全量压缩路径内的 compactConversation（src/services/compact/compact.ts 约 L387），生成对话摘要。

出站最后一公里：API 规范化 (normalizeMessagesForAPI)

经历了 L1~L3 复杂的语义与上下文容量整形后，messages 数组在逻辑上已经完美，但在物理上却未必符合各个大模型云服务商（如 Bedrock, Vertex, 1P API）的严格格式要求。

因此，在发起网络请求前的最后一刻，调用 normalizeMessagesForAPI（src/utils/messages.ts 约 L1989）进行“供应商级别的合规清洗”。流程中包含 reorderAttachmentsForAPI（同文件约 L1481）、对连续 user 的 mergeUserMessages（约 L2411）等步骤，与下述流程图节点一一对应。

核心合规动作包含：

  1. 虚拟视图剥离：将那些只为了在 REPL 终端里显示 UI 效果（isVirtual: true）的消息剔除，不浪费模型 Token。
  2. Bedrock 等环境连续 User 合并：部分云厂商的 API 严格禁止连续出现两个 role: 'user' 的消息区块，此处需自动合并。
  3. Tool Reference 维护：如果未启用 Tool Search，则彻底清洗 tool_reference；若启用，则剔除当前 Agent 已不再持有的失效工具引用。
  4. 巨型负载截断（防重试死循环）：如果上一轮 API 报了“图片太大”或“文档过大”的物理错误（生成了 Meta 错误消息），这里会向上回溯，强行把造成错误的 image 或 document 块剥离掉，防止下一轮 Agent 带着同样的超大负载重试继续撞墙。

必须强调的是，normalizeMessagesForAPI 解决的是物理协议合法性与体积优化，它处于分类器管道之外。防止 Prompt Injection 等语义攻击的清洗任务并不在这里，而是在更早的解析器入口与更后的服务端探针中完成（详见 附录 C）。

与 02 — 两阶段分类器管道 中「分类器侧 transcript 构造」是 不同平面：normalize 服务 主会话发往云厂商的合法性与体积；分类器另有对 tool results 的剥离与长度策略。其它出站路径（如 claude.ts）还会调用 ensureToolResultPairing 等修复孤儿 tool_use / tool_result（见 src/utils/messages.ts）。
