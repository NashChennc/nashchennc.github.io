Test Driven Development

2026 Apr 25
笔记
4 min read

基于 Planner-Worker 与 TDD 的 Agent 协作流

核心思想：大脑与肢体分离。旗舰模型（Planner）负责全局架构与测试编写，非旗舰模型（Worker）在极度受限的上下文中基于 TDD 闭环执行代码。

一、 核心工作流 (D-P-E)

  1. 诊断 (Diagnose) - report.md

    • 角色: Planner (旗舰模型 / 人类)
    • 动作: 审视全局依赖和核心代码，确立架构演进方向和重构策略。
    • 产出: 宏观技术决策文档。

  2. **编排 (Plan) - todolist.md & `*.spec.ts

    • 角色: Planner (旗舰模型 / 人类)
    • 动作: 将 report.md 降维拆解为原子任务，并为每个任务编写对应的测试用例。
    • 产出: 带有明确“上下文白名单”的待办列表和测试代码。

  3. 执行 (Execute) - 源码修改

    • 角色: Worker (非旗舰模型 / Code Agent)
    • 动作: 挂载局部上下文，执行 Red -> Green -> Refactor 循环。测试通过则打钩销账；遇阻则立即抛出 [BLOCKED]，拒绝越界修改。
    • 产出: 绿色的 CI 状态和原子化的 Git Commit。

二、 手动操作与 Prompt 组装指南

在实际工程中，完全自动化的全自动 Agent 往往容易失控。推荐采用 **“人类/脚本串联 + 旗舰大模型规划 + 廉价模型执行”**的半自动流水线。

阶段一：生成全局诊断 (调用旗舰模型)

操作：将项目的目录结构和核心配置文件喂给顶级模型（如 Claude 3.5 Sonnet / GPT-4o），生成 report.md。

阶段二：任务切片与 TDD 契约生成 (调用旗舰模型)

操作：基于上一步的报告，让顶级模型输出 todolist.md 的增量内容，并强制要求它先写测试。

阶段三：局部挂载与 Worker 执行 (调用非旗舰/开源模型)

操作：通过脚本提取 todolist.md 中的单条任务，拼接对应的源码和测试报错信息，喂给廉价且速度快的模型（如 Gemini Flash / Llama 3 等）。
