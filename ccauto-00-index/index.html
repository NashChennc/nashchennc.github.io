Auto Mode 安全架构分析报告系列

本系列报告基于 Claude Code 代码库，深入分析 auto mode 的安全架构实现。Auto mode 是一种将工具调用审批委托给模型分类器的权限模式——在手动逐条审批与完全跳过审批之间寻找平衡点。

报告索引

  • 01 - Entry Point 总览：权限模式体系、三层状态管理、入口矩阵、首次同意弹窗
  • 02 - 两阶段分类器管道：三层判定优先级、Stage 1/2 设计、transcript 构建、缓存策略
  • 03 - 权限剥离与信任边界：危险规则过滤、stash/restore 机制、敏感路径安全检查
  • 04 - 多 Agent 安全交接：Outbound/Inbound 分类差异、子进程继承、AFK header latch
  • 05 - 门禁、熔断与异步踢出：异步门禁校验、熔断器、kickout 变换、模型支持判定
  • 06 - 配置体系与可定制 Slot：allow/soft_deny/environment 三 Slot、设置合并、CLI 子命令
  • 07 - Deny-and-Continue 韧性设计：deny 循环、denial tracking 阈值、fail-closed/open
  • 附录 A - 单次请求的上下文整形流水线：queryLoop 出站顺序、budget/snip/microcompact/collapse/autocompact、normalizeMessagesForAPI、Harness 边界
  • 附录 B - 长会话压缩与多线程状态隔离：autoCompactIfNeeded、compactConversation、Session Memory、PTL、cached MC 隔离、Teammate 沙箱
  • 附录 C - 分类器边界之外的纵深防御：subprocessEnv、输入净化、通道 safetyCheck、系统提示、纵深时间轴、特性/env 全集速查

术语表

  • auto mode：将工具调用审批委托给 AI 分类器的权限模式，代码中也称 yolo / TRANSCRIPT_CLASSIFIER
  • yoloClassifier：auto mode 的分类器实现，通过 sideQuery 调用 Sonnet 模型，判定工具调用的安全性
  • transcript classifier：yoloClassifier 的产品名称，强调其基于对话 transcript 做判定
  • circuit breaker：熔断器，tengu_auto_mode_config.enabled === 'disabled' 时激活（参数 enabled 表示远端总开关状态），阻止进入 auto mode
  • AFK header：AFK_MODE_BETA_HEADER，auto mode 激活后会话级 latch 到 API 请求，触发服务端 PI probes
  • PI probe：Prompt Injection probe，服务端扫描工具输出中的注入攻击，本仓库通过 AFK header 触发
  • sideQuery：独立于主对话循环的 API 调用，分类器通过此通道运行，不计入主会话 token 统计
  • GrowthBook：开源的特性开关 / A/B 测试平台（growthbook.io），Anthropic 用它作为远端配置下发服务。服务端改一个值，客户端下次拉取即生效，不需要发版。核心用途是事故响应：发现安全问题时可在几秒内远端将 enabled 设为 disabled，全球客户端自动熔断 auto mode。本仓库中主要配置项为 tengu_auto_mode_config
  • strippedDangerousRules：进入 auto 时被暂存的危险 allow 规则（如 Bash(python:*)），退出时恢复
  • denial tracking：分类器拒绝计数，连续 ≥3 或总计 ≥20 时 fallback 到手动审批或终止
  • fail-closed：分类器不可用时默认拒绝（tengu_iron_gate_closed 控制；参数语义是“铁门关闭时宁可误拒绝”）
  • fail-open：分类器不可用时回退到常规权限流程（手动审批）
  • microcompact：在完整 compact 之前对 tool results 做的轻量收缩；含时间触发清理与（主线程限定的）cached microcompact（cache editing）
  • subprocess env scrub：subprocessEnv 在 CI/不可信工作流场景下从子进程环境中剥离密钥等变量，防 shell 展开外泄
