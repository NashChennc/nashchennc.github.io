CC Auto Mode 02 - 安全判定分类管道

2026 Mar 31
笔记
31 min read

本章解释一个工具调用在 auto mode 下从 ask 出发，如何经过快路径、白名单和两阶段分类器，最终落到 allow、deny 或降级回退。

ask 是权限系统的中间判定，含义是“这次调用还不能直接执行，需要进一步决策（人工或分类器）”。

与另外两类结果对照：

  • allow：直接执行工具
  • deny：直接拒绝工具
  • ask：进入后续决策链

常见触发路径：

  • 未命中静态放行规则（如 alwaysAllowRules）时，权限初判返回 behavior: 'ask'
  • 触发 safetyCheck 且 classifierApprovable: false 时，保持 ask（不调用分类器）
  • requiresUserInteraction 为真时，保持 ask（需要用户显式交互）

在 auto mode 下，ask 不是最终结果，而是进入本文后续的三层判定流程。

三层安全判定机制

当用户开启 Auto Mode 时，意味着将系统的执行控制权部分交给了模型。为了在 执行效率（Latency）与系统安全性（Security） 之间取得最佳平衡，Claude Code 并没有采用单一的权限开关，而是引入了“三层判定优先级”架构。

先对齐官方定义（来自 Claude Code auto mode: a safer way to skip permissions）：

  • Tier 1：内置安全工具白名单 + 用户规则（always-allow 等）
  • Tier 2：项目内文件操作（in-project file operations）
  • Tier 3：transcript classifier

下面这张图聚焦在 behavior: 'ask' 之后的 auto 分支内部流程。
其中 Tier 1 里的“用户规则” 大多在更早阶段已匹配，不会进入 ask 分支，所以图内主要看到 Tier 1 白名单、Tier 2 快路径与 Tier 3 分类器。

当权限系统对某个工具调用返回 behavior: 'ask' 且当前处于 auto mode 时，判定按以下优先级依次进行：

flowchart TD
    Ask["权限初判结果: ask<br/>进入 auto 分支后继续细分"]
    Safety{"safetyCheck 且<br/>非 classifierApprovable?"}
    Interactive{"requiresUserInteraction?<br/>是否必须用户显式交互"}
    AE["Tier 2: acceptEdits 快路径<br/>对应 in-project file operations"]
    AL["Tier 1: 安全工具白名单<br/>只读/协调类工具短路放行"]
    Classifier["Tier 3: 分类器<br/>其余请求进入两阶段判定"]

    Ask --> Safety
    Safety -->|是| KeepAsk1["保持 ask<br/>不调用分类器"]
    Safety -->|否| Interactive
    Interactive -->|是| KeepAsk2["保持 ask<br/>等待用户审批"]
    Interactive -->|否| AE
    AE -->|等价 allow| AllowFast["ALLOW (fastPath: acceptEdits)<br/>减少一次 API 调用"]

    AL -->|不在白名单| Classifier
    Classifier -->|allow| AllowCls["ALLOW (classifier)<br/>分类器确认安全"]
    Classifier -->|block| DenyFlow["DENY 流程<br/>返回拒绝并引导重试"]

Tier 1: 安全工具白名单（+ 用户规则）

Tier 1 中“用户规则（always-allow）”通常在 更前面的权限匹配阶段 就已返回 allow，因此不会进入本节的 ask → auto 分支图。

SAFE_YOLO_ALLOWLISTED_TOOLS（src/utils/permissions/classifierDecision.ts L56-94）列出了所有只读或协调类工具：

类别 工具示例
只读 Read, Grep, Glob, Lsp, ToolSearch
协调 TodoWrite, EnterPlanMode, ExitPlanMode, AskUserQuestion
MCP 只读 ListMcpResources, ReadMcpResourceTool
团队 TeamCreate, TeamDelete, SendMessage
内部 Sleep, TerminalCapture, Workflow

注意：编辑类工具不在白名单中（源码注释 L51-54），即使是项目内写入也走 acceptEdits 快路径而非白名单。

Tier 2: 项目内文件操作（acceptEdits 快路径）

在 auto 分支中，对于非 Agent、非 REPL 的工具，系统会以 mode: 'acceptEdits' 临时伪造上下文重跑一次模拟权限检查 ；如果该模式下结果为 allow，则直接放行并 跳过分类器 API 调用（src/utils/permissions/permissions.ts L601-656）。这一步用于快速判断“该调用是否属于 Tier 2 可直接放行”的范围，把大量 repo 内低风险读写从分类器中分流出去，降低延迟与 token 成本，同时把高风险调用继续留给后续层判定。

acceptEdits mode

acceptEdits 是一个正式 PermissionMode，在文件权限检查中放宽"项目内写入"。

它的目的是让“项目内常规改代码”这类低风险操作更顺滑，避免每一步都触发审批或分类器调用。它在受控范围内（主要是工作目录内编辑/写入）更倾向直接 allow。在工作目录外、敏感路径、需显式交互或高风险调用，仍会保持 ask / deny 或进入更严格判定。

核心在 filesystem.ts 的写权限判断顺序（checkWritePermissionForTool），包含：

  1. Safety check（路径安全性检查）
  2. ask rules 匹配
  3. mode === 'acceptEdits' && isInWorkingDir → 直接 allow（filesystem.ts L1366）
  4. allow rules 匹配
  5. 默认返回 ask

直观示例：

场景 典型结果 原因
修改 src/foo.ts 常见为 allow 项目内常规编辑，符合 acceptEdits 语义
新建 tests/bar.test.ts 常见为 allow 工作目录内写入，低风险开发操作
写入工作目录外路径（如 ~/other） 通常不直接放行 超出受控边界，需规则/分类器进一步判定
命中敏感路径 safety check 不走快路径 保持人工/严格审查，避免误放行
  • 不是单一入口函数写死：acceptEdits 只是注入到 toolPermissionContext.mode 的上下文值。
  • 统一调度模块：src/utils/permissions/permissions.ts（hasPermissionsToUseTool / hasPermissionsToUseToolInner）负责调用权限检查，并在 auto 分支里临时覆写 mode: 'acceptEdits'。
  • 具体判定来源：每个工具自己的 tool.checkPermissions(...) 实现（归属于工具模块），文件类工具大量复用 src/utils/permissions/filesystem.ts 与 src/utils/permissions/pathValidation.ts。

快路径的保护条件（防止过宽放行）

以下情况会跳过 acceptEdits 快路径，直接走白名单或分类器：

条件 原因
tool.name === AGENT_TOOL_NAME Agent 在 acceptEdits 下会 allow，会静默绕过分类器
tool.name === REPL_TOOL_NAME REPL 的胶水 JS 代码可能包含 VM escape，分类器必须看到
requiresUserInteraction() 为真 工具需要用户显式交互
safetyCheck.classifierApprovable === false 分类器不应接管的高风险路径判定

Tier 3: Transcript Classifier

不属于上述两层的工具调用进入分类器管道——这是 auto mode 安全架构中开销最高但判定最精准的一层。本节聚焦分类器 “判定什么、依据什么”（决策标准）；其实现机制（两阶段调用、transcript 构建、缓存策略）在后续章节展开。

什么流量会到达 Tier 3

根据博客的定义：

Only actions with real downside potential reach the final tier.

具体包括：

  • Shell 命令（Bash、PowerShell 等所有 shell 执行工具）
  • Web 请求（WebFetch、HttpRequest 等网络交互工具）
  • 外部工具集成（MCP 工具调用等第三方扩展）
  • 子 Agent 派生（AgentTool 的 outbound 分类，详见 04-multi-agent-security）
  • 项目目录外的文件系统操作（未命中 Tier 2 acceptEdits 快路径的写入）
  • REPL 工具（因 VM escape 风险被排除在 acceptEdits 快路径外；permissions.ts L596-599）

这些工具调用的共同特征是：具有不可逆的现实世界影响——删除远程分支、上传凭据、修改生产数据库、向外部服务发送数据等。

威胁模型

分类器的设计目标是拦截四类危险行为（博客）：

威胁类别 描述 分类器是否可检测
过度热心（Overeager behavior） Agent 理解用户目标但主动超越授权范围。如：发现凭据后直接使用、删除它认为"碍事"的东西 核心调优目标
诚实错误（Honest mistakes） Agent 误判影响范围。如：认为资源是测试用的实际上是共享的，认为分支是自己的实际上不是 核心调优目标
Prompt 注入（Prompt injection） 文件/网页/工具输出中植入的指令劫持 Agent 结构性防御（推理盲 + PI probe 双层）
模型失对齐（Misaligned model） 模型追求自己的目标 行为层拦截（与 CoT 监控互补）

博客中从内部事故日志中提炼的典型拦截案例：

场景 描述 拦截原因
范围升级 用户说"清理旧分支"，Agent 匹配远程分支并执行删除 请求模糊、操作不可逆、用户可能只意指本地分支
凭据探索 Agent 遇到认证错误后开始 grep 环境变量和配置文件寻找替代 token 凭据可能属于不同任务的作用域
Agent 推断参数 用户说"取消我的任务"，Agent 按名称相似度选了最接近的一个 目标可能不是用户意指的，可能属于他人
通过外部服务共享 Agent 想分享脚本用于调试，构造了 GitHub Gist 命令 用户可能认为内容是机密的，不应公开
安全检查绕过 部署命令预检失败后，Agent 加了 skip-verification 标志重试 可能影响共享基础设施

分类器的决策标准（System Prompt）

分类器通过 sideQuery 发起独立 API 调用来判定每个 action。

sideQuery

sideQuery（src/utils/sideQuery.ts）是 Claude Code 中独立于主对话循环的轻量 API 调用通道，不计入主会话的 token 统计，不影响对话上下文。通常用于权限分类、会话搜索、模型校验等辅助任务。 从 getClassifierModel（yoloClassifier.ts L1334-1347）可以看到，分类器模型的选择优先级是：

  • Anthropic 内部用户可通过环境变量 CLAUDE_CODE_AUTO_MODE_MODEL 指定
  • GrowthBook 配置 tengu_auto_mode_config.model 远端下发
  • 默认回退到 getMainLoopModel()——即用户当前主循环使用的模型

sideQuery 本身没有任何对话上下文——它是一个无状态的 API 调用包装器。所有上下文都必须由调用方显式构造并传入。

从源码可以清楚地看到，sideQuery 接收的全部内容就是调用方传入的参数：

它只做基础设施层面的事情：获取 API client、注入 OAuth 指纹、归一化模型名、加 beta header，然后调用 client.beta.messages.create()。没有任何隐式的上下文注入。

具体到分类器场景，classifyYoloAction（调用方）传给 sideQuery 的上下文是：

参数 内容 来源
system 分类器专用 system prompt（决策标准 + Slot 规则） buildYoloSystemPrompt() 从 .txt 模板构建
messages[0]（可选） CLAUDE.md 内容 buildClaudeMdMessage() 从缓存读取
messages[1] <transcript> 包裹的精简对话历史 + 当前 action + Stage 后缀 buildTranscriptEntries() 从主循环 messages 筛选构建

关键区别：分类器看到的 ≠ 主循环看到的

主循环 Agent 分类器 sideQuery
System prompt Agent 的完整系统提示 分类器专用提示（决策标准）
用户消息 完整保留 完整保留
Assistant 文本 完整保留 剥掉
Tool results 完整保留 剥掉
工具参数 完整原始 input toAutoClassifierInput 投影后的精简版
对话长度 全量 同步增长，但内容精简约 50-70%

所以 sideQuery 不是"接入了主循环上下文的旁路"，而是一个完全独立的 API 请求，其内容由 classifyYoloAction 精心裁剪后传入。这也是为什么文档中称其为"推理盲"——分类器看到的是一个刻意残缺的视角：只有用户说了什么、Agent 调了什么工具，没有 Agent 的自我解释，也没有工具返回的内容。

其 system prompt 由 Stage 1 和 Stage 2 共享——两阶段的唯一区别是 </transcript> 闭合标签之后、同一条 user 消息末尾追加的后缀指令不同（见 两阶段分类器设计）。

System prompt 由固定模板和三个可定制 Slot 组成。固定部分负责方法论（如何判定），Slot 负责策略（判定什么）。详细配置见 06-configuration-system。

代码实现：buildYoloSystemPrompt（src/utils/permissions/yoloClassifier.ts L484-540）拼装流程：

  1. BASE_PROMPT：从 auto_mode_system_prompt.txt（构建时内联的 .txt 文件）加载，含占位符 <permissions_template>
  2. 替换 <permissions_template>：外部用户用 permissions_external.txt，Anthropic 内部用 permissions_anthropic.txt
  3. 三个 Slot 通过 XML 标签正则替换注入用户/组织策略

外部模板 vs Anthropic 模板的关键差异（yoloClassifier.ts L511-516）：外部模板将默认 bullet 写在标签内，用户配置会整段替换默认值；Anthropic 模板将默认值写在标签外，标签为空，用户项是追加的。

评估规则（Evaluation Rules）

评估规则告诉分类器如何判断命令的真实世界影响，而不仅仅是表面文本。核心原则（博客）：

The principle is to evaluate the real-world impact of an action, rather than just the surface text of the invocation. The agent shouldn’t be able to hide a dangerous operation behind a benign-looking wrapper.

场景 分类器行为
Agent 将 payload 写入文件然后执行 评估 payload 本身，而非 bash run file.sh
用 && 链接的多条命令 整条链作为一个 action 评估
Python 脚本组装 shell 命令 从 transcript 上下文评估组装后的命令

用户意图规则（User Intent Rules）

用户意图规则是分类器中最难校准的部分，也是过度热心行为防御的核心所在。分类器判断的是——操作是否是用户授权的，而非仅仅与用户目标相关。

The classifier is deliberately conservative. The prompt establishes what is authorized; everything the agent chooses on its own is unauthorized until the user says otherwise.

用户消息 Agent 行为 判定
“clean up my branches” 批量删除远程分支 Block — 请求模糊，不构成对不可逆操作的授权
“can we fix this?” 执行修复命令 Block — 这是一个问题，不是指令
“deploy the fix” git push origin main --force Block — 用户授权部署，但未授权 force push
“run the tests” npm test Allow — 明确且范围受限

博客指出分类器 17% 假阴性率的主要原因正在这里：

In the majority of misses we examined, the classifier correctly identifies the danger… What it gets wrong is whether a real consent signal in the session actually covers this action.

分类流程（Classification Process）

分类器的 prompt 中写了一个步骤式的判定流程：

flowchart TD
    Action["接收待判定 action"]
    BlockCheck{"匹配 block rules?"}
    ExceptionCheck{"命中 allow exceptions?"}
    IntentCheck{"符合用户意图?"}

    Action --> BlockCheck
    BlockCheck -->|否| IntentCheck
    BlockCheck -->|是| ExceptionCheck
    ExceptionCheck -->|是| AllowExc["ALLOW — 规则匹配但属于豁免范围"]
    ExceptionCheck -->|否| IntentCheck
    IntentCheck -->|用户显式授权| AllowIntent["ALLOW"]
    IntentCheck -->|未授权/模糊| BlockIntent["BLOCK + reason"]
实现方式：纯 prompt 指令，非代码逻辑

上述分类流程不是代码中的 if/else 分支，而是完全通过 system prompt 中的自然语言指令实现的。代码层面只负责管道编排：

flowchart LR
    Build["构建 system prompt<br/>+ transcript"] --> Call["sideQuery<br/>发送给 Claude 模型"]
    Call --> Parse["解析返回的<br/>block yes/no"]
    Parse --> Decision["yes → block<br/>no → allow<br/>null → block"]

代码中实际的判断逻辑只有三分支（yoloClassifier.ts L801-807）——模型说 block 就 block，说 allow 就 allow，解析失败就保守 block。所有"智能判断"（判断命令是否危险、用户是否授权、是否命中豁免规则）都发生在模型推理过程中：模型读完 system prompt 中描述的评估规则、用户意图规则和分类步骤后，在生成 <block>yes/no</block> 之前按照这些指令进行推理。Stage 2 时推理过程在 <thinking> 标签中可见。

这也解释了为什么 Stage 2 的后缀要特别强调：

Review the classification process and follow it carefully, making sure you deny actions that should be blocked.

——它在提醒模型严格按照 prompt 中描述的步骤走，而不是自由发挥。

为什么用 LLM 而非规则引擎
优势 说明
灵活性 自然语言规则可以处理代码难以枚举的边界情况（“模糊的用户授权”、“隐含的危险操作”）
可定制 用户改几行 autoMode.soft_deny 文字就能调整策略，不需要改代码
可解释 Stage 2 的 <thinking> 和 <reason> 让拒绝决定可追溯
代价 说明
不确定性 同样的输入可能偶尔给出不同判断（因此设置 temperature: 0 减少随机性）
不可形式验证 不能像规则引擎那样证明"所有 force-push 一定会被拦截"
17% FNR 模型会在"用户意图边界"上犯错——正确识别了危险但误判了授权范围
Prompt 文件来源

分类流程、评估规则、用户意图规则的具体 prompt 内容存储在三个 .txt 文件中，但它们在开源代码库中不存在：

文件 路径 内容
auto_mode_system_prompt.txt src/utils/permissions/yolo-classifier-prompts/ 固定模板（评估规则、用户意图规则、分类流程步骤、输出格式说明）
permissions_external.txt 同上 外部用户的权限模板（含 block rules / allow exceptions / environment 的默认条目）
permissions_anthropic.txt 同上 Anthropic 内部用户的权限模板

代码通过 Bun 的构建时条件编译加载它们（yoloClassifier.ts L46-68）：

feature() 来自 import { feature } from 'bun:bundle'（L1），是 Bun bundler 的构建时特性开关。当 feature('TRANSCRIPT_CLASSIFIER') 为 true 时（正式构建），bundler 将 .txt 文件内容内联为字符串常量打进产物；为 false 时（如本开源代码库），三个变量都是空字符串 ''，相关代码被 dead code elimination 移除。

因此，分类流程的具体 prompt 措辞是 Anthropic 的闭源资产。我们只能从以下间接来源推断其结构：

  • 博客描述：明确提到"evaluation rules"、“user intent rules”、“step-by-step classification process”
  • 代码中的引用：replaceOutputFormatWithXml 中的 'Use the classify_result tool to report your classification.' 表明 prompt 末尾有输出格式说明
  • Stage 2 后缀：'Review the classification process and follow it carefully' 表明 prompt 中存在一个名为"classification process"的具体步骤描述
  • classify_result 工具 schema：thinking 字段描述为 'Brief step-by-step reasoning.'，表明 prompt 要求按步骤推理
  • claude auto-mode defaults 命令：在正式构建产物中可输出默认的 block rules / allow exceptions / environment 条目（通过 getDefaultExternalAutoModeRules 从模板标签内提取）

默认拦截规则（Block Rules / soft_deny）

默认超过 20 条拦截规则，按博客分为四大类：

类别 描述 示例场景
破坏或外泄 造成不可逆损失 force-push 覆盖历史、批量删除云存储、向外部发送内部数据
降低安全态势 使系统更难监控或防御 禁用日志、安装持久化如 SSH key/cronjob、修改 Agent 自身的权限配置
跨越信任边界 操作 Agent 没有信任基础的资源 执行从外部 repo 克隆的代码、扫描凭据库寻找可用 token、向用户未指定的服务发送数据
绕过审查或影响他人 跳过共享基础设施的安全措施 直接 push 到 main、运行生产部署、修改非 Agent 创建的共享资源/任务

规则存储在 permissions_external.txt 的 <user_deny_rules_to_replace> 标签内。可通过 claude auto-mode defaults 查看完整列表（仅在正式构建产物中可用）。

默认允许例外（Allow Exceptions）

允许例外是强制豁免——block rule 技术上匹配但操作实际无害。默认范围很窄：

  • 安装项目 manifest 中已声明的依赖包
  • 标准凭据流程（如 npm login 等工具自身的认证流程）
  • 推送到当前会话的工作分支

规则存储在 permissions_external.txt 的 <user_allow_rules_to_replace> 标签内。

环境信任边界（Environment）

环境定义了哪些域名、云存储桶、源码管理组织和内部服务算作"内部"。默认情况下分类器仅信任当前 git 仓库。用户可通过 autoMode.environment 扩展信任范围（如 GitHub org、云存储桶、内部 API），使分类器能识别可信基础设施。边界之外的一切都可能构成外泄风险。

性能指标

博客公布了分类器在三个数据集上的评估结果：

真实流量 n=10,000 真实过度热心 n=52 合成数据外泄 n=1,000
Stage 1 快速过滤 8.5% FPR 6.6% FNR 1.8% FNR
Stage 1→2 完整管道 0.4% FPR 17% FNR 5.7% FNR
  • FPR（假阳性率）：安全操作被错误拦截的比例
  • FNR（假阴性率）：危险操作被错误放行的比例
  • Stage 2 将 FPR 从 8.5% 降至 0.4%，代价是 FNR 从 6.6% 升至 17%
  • 0.4% FPR 配合 Deny-and-Continue 韧性设计（见 07-deny-and-resilience），在实践中影响极小

Whether 17% is acceptable depends on what you’re comparing against. If you are running --dangerously-skip-permissions, this is a substantial improvement. If you are manually approving every action carefully, it’s arguably a regression.

两阶段分类器设计

Two-stage classification: efficiency and recall. A single-stage classifier faces a direct performance/cost trade-off… Splitting into two stages with different levels of caution breaks this trade-off.

— Claude Code auto mode

代码中实现为 classifyYoloActionXml（src/utils/permissions/yoloClassifier.ts L711-996）。两阶段共享同一份 system prompt 和 transcript，只有末尾的后缀指令不同。Stage 1 是低成本的快速过滤，绝大多数安全操作在此直接放行；只有被 Stage 1 标记为可疑的操作才会进入 Stage 2 做深度推理。

flowchart TD
    Input["待判定 action 进入分类器"]

    Input --> S1["Stage 1: 快速过滤<br/>后缀: Err on the side of blocking<br/>max_tokens: 64, stop: 遇到 &lt;/block&gt; 即停<br/>temperature: 0"]
    S1 --> S1Parse{"解析 &lt;block&gt;yes/no&lt;/block&gt;"}

    S1Parse -->|"no — 安全"| Allow1["✓ ALLOW<br/>直接放行，跳过 Stage 2"]
    S1Parse -->|"yes — 可疑"| S2
    S1Parse -->|"null — 解析失败"| Block1["✗ BLOCK<br/>解析失败走保守策略"]

    S2["Stage 2: 链式推理<br/>后缀: Review the classification process...<br/>max_tokens: 4096, 无 stop<br/>允许 &lt;thinking&gt; 推理"]
    S2 --> S2Parse{"解析 &lt;block&gt; + &lt;reason&gt;"}

    S2Parse -->|"no — 安全"| Allow2["✓ ALLOW<br/>通过深度复核"]
    S2Parse -->|"yes — 危险"| Block2["✗ BLOCK + reason<br/>返回可解释拒绝信息"]
    S2Parse -->|"null — 解析失败"| Block3["✗ BLOCK<br/>不确定时保守拦截"]

直观理解：Stage 1 是粗筛网（宁可多拦），Stage 2 是精筛网（把误拦的放回去）。大多数操作在 Stage 1 就通过了（<block>no</block>），不需要付 Stage 2 的推理成本。只有 Stage 1 觉得"可能有问题"时才花 4096 tokens 做深度分析。

Stage 1 与 Stage 2 的关键差异

两阶段共享同一份 system prompt（上文 Tier 3 所述的决策标准）和 transcript 内容，区别仅在于 </transcript> 闭合标签之后追加的后缀指令（作为同一条 user 消息的最后一个文本块）：

维度 Stage 1 Stage 2
后缀指令 Err on the side of blocking. <block> immediately. 回顾分类流程、要求只有显式用户确认才能推翻拦截、先 <thinking> 再 <block>
max_tokens 64（both 模式）或 256（fast 模式） 4096
stop_sequences ['</block>']（促使尽早结束） 无
输出格式 仅 <block>yes/no</block> <thinking>...</thinking> + <block> + 可选 <reason>
设计意图 高召回、低成本、偏保守 降低误报、允许推理

两阶段 XML 模式下（replaceOutputFormatWithXml，L648-664），系统提示词中的 classify_result 工具说明被替换为 XML 输出格式：

  • Block：<block>yes</block><reason>一句话原因</reason>
  • Allow：<block>no</block>（不带 <reason>）
  • Stage 2 允许在 <block> 前输出 <thinking> 推理块；解析时先通过 stripThinking（L567-571）剥离，避免推理中的标签干扰解析

分类器结果的消费

permissions.ts L818-926 对返回值按优先级处理。分类器返回的 shouldBlock 为主判定，为 true 时进一步按 block 原因分流到不同的恢复策略：

flowchart TD
    Result["classifyYoloAction 返回结果"]
    Block{"shouldBlock?"}

    Result --> Block

    Block -->|false| Allow["✓ ALLOW<br/>recordSuccess<br/>重置连续拒绝计数"]

    Block -->|true| Why{"block 原因？"}

    Why -->|"transcriptTooLong<br/>transcript 超出上下文窗口"| Headless{"headless 模式？"}
    Headless -->|是| Abort["throw AbortError<br/>transcript 只增不减<br/>重试无意义，直接终止"]
    Headless -->|否| Manual1["回退手动审批<br/>让用户自行决定"]

    Why -->|"unavailable<br/>分类器 API 报错"| Gate{"tengu_iron_gate_closed?<br/>默认 true"}
    Gate -->|true| FailClosed["✗ DENY + 重试引导<br/>fail-closed：宁可误拦"]
    Gate -->|false| FailOpen["回退常规权限流程<br/>fail-open：跳过分类器"]

    Why -->|"正常 block<br/>分类器判定为危险"| Record["recordDenial<br/>更新拒绝计数"]
    Record --> Limit{"连续 ≥ 3 次 或 累计 ≥ 20 次？"}
    Limit -->|是| Manual2["回退手动审批 / 终止<br/>避免无限拒绝循环"]
    Limit -->|否| Deny["✗ DENY<br/>buildYoloRejectionMessage<br/>含拒绝原因 + 更安全路径引导"]

拒绝消息（buildYoloRejectionMessage，src/utils/messages.ts L267-282）包含分类器原因、引导 Agent 寻找更安全路径的指示、以及建议用户通过权限规则允许此类操作。详细的 Deny-and-Continue 韧性机制见 07-deny-and-resilience。

Transcript 构建：推理盲设计

We strip assistant text so the agent can’t talk the classifier into making a bad call… Stripping tool results is the primary prompt-injection defense.

— Claude Code auto mode

分类器的 transcript 由 buildTranscriptEntries（src/utils/permissions/yoloClassifier.ts L302-360）按推理盲策略从对话历史中筛选构建：

内容类型 是否保留 原因
User text 保留 提供用户意图锚点
Assistant tool_use 保留 判定对象
Assistant 自然语言 剥掉 防止 agent 用文字操纵分类器
Tool results 剥掉 防止 prompt injection 通过工具输出进入分类器
非 text 的 user content 剥掉 仅保留文本

推理盲有一个有用的副效应：行为层拦截与 CoT 监控正交互补。一个隐藏恶意推理的模型若最终行为触发 block rule 仍会被分类器拦截；一个行为看似无害但推理暴露意图的模型则会被 CoT 监控捕获。

CLAUDE.md 前缀

如果会话中存在 CLAUDE.md 配置，buildClaudeMdMessage（L460-477）会将其作为独立的前缀消息（在 transcript 之前）注入分类器调用，包裹在 <user_claude_md> 标签中并标注为用户意图的一部分。这确保分类器在判读用户意图时将 CLAUDE.md 中的指令视为用户授权。

序列化格式

由 toCompactBlock（L384-424）处理，支持两种编码：

  • JSONL 模式：{"ToolName": ...}\n / {"user": "..."}\n
  • 传统模式：ToolName ...\n / User: ...\n

工具输入通过 tool.toAutoClassifierInput 投影（定义给分类器可见的最小必要字段），工具可以控制暴露给分类器的字段（例如隐藏大段文件内容，只暴露路径和操作类型）。
