CC Auto Mode 附录 B - 长会话压缩与多线程状态隔离

2026 Mar 31
笔记
14 min read

在附录 A 中，我们探讨了出站前轻量级的上下文整形（Shaping）。然而，当一个 Agent 彻夜重构代码，积累了上百轮对话后，轻量级的裁剪终将杯水车薪。此时必须启动真正的全量压缩（Compaction）：让模型阅读前半段历史，生成高度浓缩的摘要，并截断旧记录。

这里的核心技术挑战并非“如何写 Prompt 让模型总结”，而是在一套具备主进程、后台联想（Prompt Suggestion）、子 Agent（Teammate）并行运行的架构中，如何保证压缩操作安全、隔离且不引发雪崩。

1. 终极防线：全量压缩 (autoCompactIfNeeded)

全量压缩不是毫无顾忌地触发的，它是一项极其昂贵的操作。触发阈值通常被设定为当前模型有效上下文窗口减去一个安全缓冲带（AUTOCOMPACT_BUFFER_TOKENS，约 13,000 Tokens；常量定义见 src/services/compact/autoCompact.ts 约 L62，阈值计算见同文件 getAutoCompactThreshold 约 L72 起）。

进入 autoCompactIfNeeded（src/services/compact/autoCompact.ts 约 L241）后，顺序为：先检查 DISABLE_COMPACT（约 L253–255）；再检查 连续失败熔断 tracking.consecutiveFailures >= MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES（值为 3，约 L67–70、L257–265）；再通过内部 shouldAutoCompact（同文件 shouldAutoCompact / calculateTokenWarningState 一带，约 L225–238）判断当前估算 token 是否真需压缩——未过阈则直接返回，不会进入 Session Memory 或 compactConversation。

若判定需要压缩，则先尝试 trySessionMemoryCompaction（约 L287–292；实现 src/services/compact/sessionMemoryCompact.ts 约 L514）。若返回非空：视为已压缩，执行 setLastSummarizedMessageId(undefined)、runPostCompactCleanup(querySource)；在 PROMPT_CACHE_BREAK_DETECTION 开启时还会 notifyCompaction（约 L302–304），避免 cache break 误报。否则再 await compactConversation(...)（约 L312–321）。成功时 consecutiveFailures 归零；catch 中非用户中止的错误递增失败计数，达阈打熔断日志（约 L334–349）。用户中止（ERROR_MESSAGE_USER_ABORT）不记入失败链（catch 内过滤）。

1.2 摘要生成中的 PTL（Prompt Too Long）重试自愈

在 compactConversation 内部生成摘要时，系统面临一个“第 22 条军规”式的困境：用来生成摘要的 Prompt 本身，可能就已经超出了模型的 Context 限制（即报 prompt_too_long 错误）。主实现位于 src/services/compact/compact.ts（compactConversation 约 L387）。

Claude Code 在这里实现了一个非常优雅的退避与自愈循环：如果 API 报错 PTL，它不会直接放弃，而是通过 truncateHeadForPTLRetry（同文件约 L243）强行切掉最早期的一小段历史，再次重试；摘要流由 streamCompactSummary（同文件约 L1136）发起，直到成功或达到最大重试次数。

flowchart TD
  loopHead[for 循环: 尝试生成摘要]
  streamCall["streamCompactSummary<br/>流式拉取压缩摘要"]
  
  isPtl{返回结果是否以<br/>PTL 错误前缀开头?}
  leaveLoop[成功生成摘要，跳出循环]
  
  ptlIncr[重试次数 ptlAttempts +1]
  retryOk{是否超最大重试次数<br/>且还能截断头部?}
  truncate["truncateHeadForPTLRetry<br/>截掉最老一段再重试"]
  throwPtl[抛出严重错误<br/>PROMPT_TOO_LONG]

  streamCall --> isPtl

  retryOk -->|是| truncate --> loopHead
  retryOk -->|否| throwPtl

调窗与测试用环境变量（非默认用户配置）：

  • CLAUDE_CODE_AUTO_COMPACT_WINDOW：截断有效上下文窗口上限（getEffectiveContextWindowSize，约 L40–45）。
  • CLAUDE_AUTOCOMPACT_PCT_OVERRIDE：按有效窗口百分比覆盖 autocompact 阈值（约 L79–87）。
  • DISABLE_COMPACT：关闭 autoCompactIfNeeded 入口（约 L253–255）。

与 Harness design 的 compaction 叙事一致：在单会话内延续任务。与 07 — Deny-and-Continue 中「transcript 超过分类器窗口」的降级对照——压缩缓解膨胀，但不能保证分类器窗口永不被突破。

2. 核心隔离架构：Microcompact 与全局缓存陷阱

相比于重型的全量总结，microcompact（在 附录 A 提及）是一种极其轻量、高频触发的缓存编辑与工具结果清理机制。对外入口为 microcompactMessages（src/services/compact/microCompact.ts 约 L253）。实现上的顺序要点如下：

  1. 时间优先：若距上一条 assistant 消息间隔超过阈值，认为服务端 cache 已 冷，先走 maybeTimeBasedMicrocompact（约 L261–266）清理旧 tool results；此路径 跳过 cached microcompact（缓存编辑假设 warm cache）。
  2. Cached microcompact：特性、模型与 主线程 querySource 均满足时走 cachedMicrocompactPath（约 L276–285）。
  3. 仅主线程：fork（如 session_memory、prompt_suggestion）不得把 tool_results 登记进全局 cachedMCState（同文件 L272–275 注释）。

Claude Code 深度利用了 Anthropic API 的 Prompt Cache Editing 能力（通过传入 cache_edits 来按 tool_use_id 精准摘除缓存树中的无用大块文本，而不必重写整个前缀）。模块级全局状态 cachedMCState 与 ensureCachedMCState / getCachedMCModule 等同文件约 L56–80；仅主线程在 isMainThreadSource(querySource) 为真且特性开启时才进入 cachedMicrocompactPath（同文件约 L249–251、L272–284、L305）。

然而，这种对远端 API 缓存状态的操作，在客户端引入了一个巨大的并发隐患。

跨线程缓存污染

假设主线程正在与用户正常对话。此时，后台静默启动了一个 Fork 线程（例如 prompt_suggestion，用于预测用户下一步指令）。 如果这个 Fork 线程在执行时，也去调用了 Cached Microcompact，它就会把只在 Fork 会话里存在的 tool_use_id 登记到客户端的全局缓存状态表 (cachedMCState) 中。

结果灾难性：主线程在下一次发起对话时，读取了被污染的全局表，并向 API 发送指令：“请从缓存中删掉工具 ID: XYZ”。API 会直接报错崩溃，因为在主对话的真实上下文里，根本不存在 XYZ 这个工具调用！

为了解决这个问题，系统强制实行了基于 QuerySource 的严格路径分流。

2.1 主线程与 Fork 线程的决断网络

flowchart TB
  entry["microcompactMessages<br/>microcompact 对外入口"]
  timeCheck{距离上一次请求<br/>是否间隔过久?}
  
  timeCheck -->|"是 (Cache 已冷)"| timePath["maybeTimeBasedMicrocompact<br/>cache 冷则先时间触发清理"]
  timeCheck -->|"否 (Cache 热)"| route{判断 QuerySource<br/>是否为 'repl_main_thread*'}
  
  route -->|"是 (主线程)"| gate{开启了 Cached MC 特性?}
  route -->|"否 (Fork 分支)"| forkBlock[禁止进入 Cached 路径]
  forkBlock --> forkWhy[为防止主线程 API 崩溃<br/>绝对禁止向全局状态表注册私有 ID]
  
  gate -->|"通过"| cached["cachedMicrocompactPath<br/>cache_edits 摘掉 tool results"]
  gate -->|"未开启"| noop[原样返回 messages]
  
  cached --> gmc["cachedMCState<br/>进程内全局缓存编辑状态"]
  
  timePath --> exit[返回出站 Messages]
  forkWhy --> exit
  gmc --> exit
  noop --> exit

设计精髓：这是一种经典的多源并发防撞设计。只有主线程拥有全局抽象状态的“写权限”，所有临时创建的影子线程（Session Memory, 后台 Hook）只能使用不依赖缓存编辑的退化模式，从而物理隔绝了上下文状态污染的可能性。安全/正确性含义：这是进程内 全局抽象状态与多 QuerySource 并发写 的一致性问题；若做错，主线程与 API cache 状态错位会导致异常或意外前缀重写。

2.2 延迟的边界确认 (Deferred Boundary)

对于主线程，它决定向 API 发送 Cache Edits 请求后，不会立即在本地客户端更新自己的数据结构。它必须等待 API 流式响应完全结束，并且确认 API 返回的 cache_deleted_input_tokens 确实产生了收益（减去基线后 Delta > 0），才会在本地 yield 一条 createMicrocompactBoundaryMessage（消息构造见 src/utils/messages.ts 约 L4557；在 query 成功路径上产出见 src/query.ts 约 L870–890，与 pendingCacheEdits 基线相减逻辑一致），完成本地状态机的闭环。

为何 boundary 要等流结束？ API 的 cache_deleted_input_tokens 为 累计/粘性 值，需用 microcompact 返回前记录的 baseline 相减得到 本轮 delta；仅当 delta > 0 才值得写入边界消息（实现见 query.ts 约 L420–425 与 L866–891）。

sequenceDiagram
  participant Q as "queryLoop<br/>主循环"
  participant M as "microcompact<br/>轻量收缩"
  participant API as "queryModelStream<br/>流式请求模型"
  Q->>M: microcompact(messages)<br/>计算是否产生 pendingCacheEdits
  alt pendingCacheEdits 为空
    M-->>Q: 无 deferred edits<br/>本轮无延迟 boundary
  else pendingCacheEdits 有值
    M-->>Q: messages 与 pendingCacheEdits<br/>下请求携带 cache_edits
  end
  Q->>API: 发送本轮请求<br/>含 cache_edits 等
  API-->>Q: 流式响应与 usage<br/>含累计 cache_deleted_input_tokens
  alt CACHED_MICROCOMPACT 且 pendingCacheEdits
    Note over Q: 读 cache_deleted_input_tokens 减 baseline<br/>得到本轮删除 token delta
    opt delta 大于 0
      Q->>Q: yield createMicrocompactBoundaryMessage<br/>写入边界事件闭环
    end
  end

3. 子进程 (Teammate) 的隔离生态墙

不仅主线程与后台预测线程需要隔离，Claude Code 中的“多 Agent 协作”（主 Agent 唤起一个跑在同一进程中的子 Teammate Agent）同样面临严峻的状态隔离挑战。

当子 Teammate 在执行一个需要 50 轮迭代的复杂任务时，其内部上下文会极速膨胀并触发 compactConversation。如果此时直接套用全局的压缩逻辑，会发生两件可怕的事情：

  1. Teammate 会清空主会话辛苦建立的文件状态缓存 (readFileState)。
  2. Teammate 的后台压缩进度，会触发主会话的用户界面 (UI) 弹出压缩进度条，导致终端显示错乱。

因此，runInProcessTeammate（src/utils/swarm/inProcessRunner.ts 约 L883）在每次迭代检查到 Token 超标时，必须构造一个完全隔离的 ToolUseContext（沙箱上下文）（同文件约 L1071–1103：在 tokenCountWithEstimation 与 getAutoCompactThreshold 判定之后分支进入压缩）。

flowchart TB
  subgraph parentSession [主会话 Parent Session]
    parentRFS["readFileState<br/>主会话文件读取缓存"]
    parentUI[终端 REPL 进度条回调]
  end
  
  subgraph teammateLoop ["队友子代理（Teammate Agent）"]
    iter[执行单轮工具迭代]
    overTok["tokenCountWithEstimation<br/>估算是否超 getAutoCompactThreshold"]
    
    isoCtx["隔离 ToolUseContext<br/>cloneFileStateCache 等沙箱补丁"]
    cc["compactConversation<br/>队友内全量压缩"]
    
    iter --> overTok
    overTok -->|"是"| isoCtx
    isoCtx --> cc
    cc --> iter
    overTok -->|"否"| iter
  end
  
  isoCtx -.->|"克隆 (Clone)<br/>只读不写回"| parentRFS
  isoCtx -.->|"抹除回调<br/>静默执行"| parentUI

核心防御动作：

  • 克隆文件树 (cloneFileStateCache)：实现见 src/utils/fileStateCache.ts 约 L122；在 Teammate 压缩路径上写入隔离上下文（inProcessRunner.ts 约 L1086）。
  • 屏蔽 UI 副作用 (onCompactProgress: undefined)：同一段沙箱补丁中一并置 setStreamMode: undefined（inProcessRunner.ts 约 L1087–1088），避免子代理压缩进度回调打扰主会话 UI；压缩仍调用 compactConversation（约 L1090）与 buildPostCompactMessages（约 L1104）刷新 contextMessages。

压缩成功后还会 resetMicrocompactState（旧 tool id 失效）、必要时重置队友侧 contentReplacementState / teammateReplacementState，并 更新 task 消息与 AppState 镜像，避免长时循环中镜像无限增长（同文件约 L1105–1124）。详见 04 — 多 Agent 安全交接 中 in-process 与 teammate 相关讨论。

通过这三套严密的隔离与防熔断机制（主分支防雪崩、多线程缓存墙、子代理状态沙箱），Claude Code 构建了一个极度强韧的长程运行环境。这正是其能够在数小时的无人值守 Auto Mode 下不崩溃的底层基石。
